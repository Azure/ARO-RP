# Agentic Hints for Log Search in ARO Classic

- IMPORTANT: this document is referenced by agentic workflows, DO NOT REMOVE IT.
- Read `docs/ai/classic-debugging.md` first. Load this file only when the current task is primarily log search or request tracing from logs.
- This file assumes a Kusto/Geneva log-search skill. Do not assume access to admin APIs, `oc`, Prometheus, or live cluster actions.
- If any of the info below turns out not to be accurate, suggest an update PR at the end of the session.

## Differences from ARO HCP

- Do not assume the HCP Kusto tables (`HostedControlPlaneLogs`, `frontendLogs`, `backendLogs`, etc.) or Grafana datasource names exist in ARO Classic.
- The repo does not document a complete Classic-specific Kusto table inventory. Start with cross-family pivots like `request_id` and often `resource_id`, then narrow with family-specific fields such as `request_path`, `LOGKIND`, and known log messages instead of HCP-specific table names.
- ARO Classic logs are emitted through Geneva/MDSD. Metrics flow through MDM/statsd, but metrics are out of scope for this file.

## Start Here

- If you already have an RP response or log line for the original RP-handled request, start with `request_id`.
- If you are looking at an `Azure-AsyncOperation` polling request, pivot on the async operation ID from the polling URL/path first; each poll gets a fresh `request_id`.
- If you do not have `request_id`, start with `resource_id`; fall back to `subscription_id`, `resource_group`, `resource_name`, and a time window.
- Customer `PUT` / `PATCH` / `DELETE` cluster mutations are async. Default search order: frontend access logs -> `frontendqos` -> backend `asyncqos` -> outbound dependency logs -> optional Hive / installer logs.
- For admin or maintenance incidents that are already visible in logs, use admin route names only as `request_path` filters. These are often frontend-only log trails, so start with `request_path` plus handler-specific messages and do not assume `frontendqos` or backend `asyncqos` unless code inspection shows an async handoff. This skill cannot call those endpoints.

## High-Value Fields

- `request_id` is the best primary pivot for a single RP-handled request, and usually for the original mutation request. The RP always generates it and returns it in `X-Ms-Request-Id`.
- `correlation_id` and `client_request_id` come from caller or ARM headers and may be empty; use them opportunistically.
- `operation_id` is also generated by the RP and appears in logs, but do not confuse it with the `Azure-AsyncOperation` polling URL.
- For `Azure-AsyncOperation` status/result polling, use the async operation ID embedded in the URL/path to bridge back to the original operation; each poll has a new `request_id`.
- Inbound logs enrich from request paths, so matching paths typically produce `subscription_id`, `resource_group`, `resource_name`, and `resource_id`.
- Backend logs enrich from the cluster resource ID and can also include `cluster_deployment_namespace`, which is useful for bridging into Hive-backed flows.
- Prefer case-insensitive matching for `resource_id`, `subscription_id`, `resource_group`, `resource_name`, and `cluster_deployment_namespace`; some enrichment lowercases these fields while path-derived values can preserve incoming case.

## Log Families

- Access logs from `pkg/frontend/middleware/log.go` use the messages `read request` and `sent response`. They carry `request_method`, `request_path`, `request_user_agent`, and, on `sent response`, `response_status_code` and `duration`.
- `LOGKIND=frontendqos` in `pkg/frontend/frontend.go` marks ARM-facing operation-result logs with `operationType`, `resultType`, and, on failures, `errorDetails`. Most admin routes do not use this wrapper; for those, access logs and handler-specific log lines are the main search surface.
- `LOGKIND=asyncqos` in `pkg/backend/openshiftcluster.go` marks terminal backend results with the messages `long running operation succeeded` and `long running operation failed`. These logs add `operationType`, `resultType`, `clusterIdentity`, and, on failure, `errorDetails`.
- `HttpRequestStart` and `HttpRequestEnd` in `pkg/util/azureclient/roundtripper.go` add `LOGKIND=outboundRequests` when full outbound logging is enabled. In local dev, this path is disabled by default unless `ARO_ENABLE_OUTBOUND_HTTP_LOGGING=true`.
- `OutboundRequestFailed` and `OutboundRequestConflict` from the same logging policy are always important search terms, even when `LOGKIND=outboundRequests` is absent. They include fields like `request_host`, `request_method`, and `response_status_code`.
- For Hive-backed flows, `pkg/util/log/hive.go` annotates `ClusterDeployment` objects so Hive and installer logs can inherit `correlation_id`, `client_request_id`, `request_id`, and resource fields. The exact downstream searchable schema depends on the environment.
- Cluster logs are shipped by the Geneva logging controller in `pkg/operator/controllers/genevalogging/`. It injects `RESOURCE_ID`, `SUBSCRIPTION_ID`, `RESOURCE_GROUP`, and `RESOURCE_NAME` into the `mdsd` DaemonSet; treat these as likely scoping keys and verify the exact searchable dimensions in your environment.

## Async Mutation Flow

- ARM-facing `PUT` / `PATCH` / `DELETE` requests log through frontend access logs and often produce a `frontendqos` result log immediately.
- The frontend stores `CorrelationData` on the cluster document and returns an `Azure-AsyncOperation` polling URL. That polling URL is not the same thing as `operation_id`.
- Status and result polls use paths like `/operationsstatus/{operationId}` and `/operationresults/{operationId}`. Treat those poll requests as new requests with new `request_id` values.
- The backend later dequeues the document and emits terminal `asyncqos` logs for the long-running result.
- If the operation is Hive-backed, the same `request_id` / resource fields can also appear in Hive / installer logs through `hive.openshift.io/additional-log-fields`.

## Useful Path and Message Filters

- Use `request_path` for quick narrowing when the operation type is already known.
- For customer resource mutations, search the cluster resource path plus `read request`, `sent response`, `front end operation failed`, `front end operation succeeded`, `long running operation failed`, and `long running operation succeeded`.
- For admin or maintenance issues already implicated by logs, route suffixes such as `/preresizevalidation`, `/resizecontrolplane`, the exact `/resize` route, and `/investigate` are useful `request_path` filters. Treat these as examples, not an exhaustive list. This is log-search guidance only, not endpoint guidance.
- If a control-plane resize incident is already suspected, useful messages from `pkg/frontend/admin_openshiftcluster_resize_controlplane.go` include `Resizing control plane node`, `Cordoning node`, `Draining node`, `Stopping VM`, `Resizing VM`, `Starting VM`, `Waiting for node`, `Uncordoning node`, `Updating Machine object`, and `Updating Node labels`.

## Code Locations

- `pkg/api/correlation.go` - correlation field generation
- `pkg/util/log/log.go` - resource and correlation field enrichment
- `pkg/frontend/middleware/log.go` - inbound request/access log fields
- `pkg/frontend/frontend.go` - `frontendqos` operation-result logs
- `pkg/frontend/openshiftcluster_putorpatch.go` - async handoff for create and update
- `pkg/frontend/openshiftcluster_delete.go` - async handoff for delete
- `pkg/backend/openshiftcluster.go` - `asyncqos` terminal backend logs
- `pkg/util/azureclient/roundtripper.go` - outbound dependency logging
- `pkg/util/log/hive.go` - Hive / installer log-field propagation
- `pkg/operator/controllers/genevalogging/` - cluster log shipping via `fluentbit` and `mdsd`
